# Codebase Concerns

**Analysis Date:** 2026-04-22

## Security Considerations

**No authentication or authorisation:**
- Risk: Any network-reachable user can read all annotation data, modify classifications, assign tags, trigger LLM seeding (which costs API credits), and download the full export.
- Files: `backend/app/main.py` (CORS wildcard), all routers in `backend/app/routers/`
- Current mitigation: CORS `allow_origins=["*"]` is explicitly noted in code as "No auth yet". `allow_credentials=False` avoids the invalid combination but provides no actual protection.
- Recommendations: Add HTTP Basic Auth or API key middleware before exposing to any non-localhost network. FastAPI supports `HTTPBasic` and `APIKeyHeader` with minimal changes.

**LLM seeding is unauthenticated and unbounded:**
- Risk: Any caller can POST `/api/llm/seed` to trigger a multi-domain LLM run using the server's `ANTHROPIC_API_KEY` or `OPENROUTER_API_KEY`. With ~20 L1 domains each requiring a separate LLM call, a single POST can generate significant API spend.
- Files: `backend/app/routers/llm.py`, `backend/app/llm/seeding.py`
- Current mitigation: A 409 guard prevents two concurrent runs, but there is no rate limiting or caller authentication.
- Recommendations: Require authentication before the seed endpoint; add a per-day or per-hour rate limit.

**API keys sourced from environment only — no runtime validation at startup:**
- Risk: If `ANTHROPIC_API_KEY` or `OPENROUTER_API_KEY` are absent, the error is deferred until the first chat or seed request. Users see a 500 with a raw `RuntimeError` message that leaks the variable name.
- Files: `backend/app/llm/client.py` lines 32-34, 97-98
- Current mitigation: None at startup.
- Recommendations: Check for keys in `lifespan()` in `backend/app/main.py` and log a clear warning; return a 503 with a user-friendly message from the affected endpoints.

**Notes content is written directly to disk without sanitisation:**
- Risk: Notes are stored as raw Markdown and re-read verbatim. If the export HTML renderer ever renders notes as raw HTML (it currently escapes them via `_esc()`), stored XSS would be possible.
- Files: `backend/app/persistence.py` (`write_note`), `backend/app/routers/export.py` (`_build_markdown`, `_md_to_html`)
- Current mitigation: The HTML exporter applies `_esc()` to all user-supplied strings. The risk is low currently but will increase if the notes renderer is upgraded to full Markdown-to-HTML.

**OpenRouter HTTP-Referer header hardcoded to localhost:**
- Risk: In production, `HTTP-Referer: http://localhost:3000` is sent to OpenRouter for every request. This is a misleading signal and could cause billing/attribution issues if OpenRouter ever enforces referer-based policies.
- Files: `backend/app/llm/client.py` line 113
- Recommendations: Make the referer configurable via an environment variable.

## Tech Debt

**No write concurrency control on Markdown files:**
- Issue: Every write operation (classifications, tags, teams, notes, value streams) is a full read-parse-rewrite of the entire file with no file locking. Two concurrent POST/PUT requests can produce a lost-update race condition.
- Files: All routers in `backend/app/routers/` that call `write_md_file` or `write_note`; `backend/app/persistence.py`
- Impact: Silent data loss if two users edit different nodes simultaneously. Unlikely in current single-user deployment but a real risk if opened to a team.
- Fix approach: Use `fcntl.flock` around read-modify-write cycles in `persistence.py`, or introduce a single asyncio `Lock` per file keyed by filename.

**Global module-level cache for chat tree context is never invalidated:**
- Issue: `_tree_context_cache` in `backend/app/routers/chat.py` is set once and never cleared. The process tree is static (loaded from Excel at startup) so this is currently harmless, but if the tree were ever made dynamic the cache would serve stale data indefinitely.
- Files: `backend/app/routers/chat.py` lines 17-24
- Impact: Low today; medium if tree reloading is added.
- Fix approach: Remove the module-level global and pass the cache as app state, or accept the current design as an intentional optimisation with a comment.

**Tag slug ID generation has a collision edge case:**
- Issue: When creating a tag, the ID is generated by slugifying the name. If two different names produce the same slug (e.g. "My Tag" and "My-Tag"), the fallback is `{slug}-{len(rows)}` which is not guaranteed unique if the list is already long.
- Files: `backend/app/routers/tags.py` lines 49-52
- Impact: Low probability; would cause a duplicate-ID row in `tags.md` that silently overwrites on next read.
- Fix approach: After generating the fallback, check again for uniqueness and increment a counter until unique.

**Chat system prompt injects full process tree (~2900 nodes) on every request:**
- Issue: The tree context is cached but can still be very large. The prompt includes all ~2900 node IDs and names, which consumes a significant portion of the LLM's context window before any conversation history.
- Files: `backend/app/routers/chat.py` (`_build_tree_context`, `_get_tree_context`)
- Impact: Higher token cost per chat request; may cause context window overflow for models with smaller limits.
- Fix approach: Consider summarising to L1-L3 only, or making the tree context opt-in per query rather than always included.

**LLM seeding result counter reports `classified` but seed returns `seeded`:**
- Issue: `SeedingButton.tsx` reads `data.result?.classified` (line 46) but `seed_classifications` returns `{"seeded": N, "domains": D, "errors": [...]}`. The key is `seeded`, not `classified`, so the count always displays as 0.
- Files: `frontend/src/components/SeedingButton.tsx` line 46, `backend/app/llm/seeding.py` lines 159-163
- Impact: Cosmetic only — seeding works correctly, only the success message count is wrong.
- Fix approach: Change `data.result?.classified` to `data.result?.seeded` in `SeedingButton.tsx`.

**`process_tree` is stored on `app.state` with no type annotation:**
- Issue: `app.state.process_tree` and `app.state.seed_status` are set as dynamic attributes on the FastAPI app state object. There is no typed `AppState` class, so type checkers cannot verify access patterns.
- Files: `backend/app/main.py`, all routers that access `request.app.state`
- Impact: Low risk currently; potential for silent `AttributeError` if a router is loaded before `lifespan()` runs in tests.
- Fix approach: Define a typed `AppState` dataclass and assign it to `app.state` in `lifespan()`.

**`find_node` is a recursive O(n) tree walk with no memoisation:**
- Issue: `find_node` in `backend/app/utils.py` performs a depth-first walk of the entire process tree (~2900 nodes) for every node lookup. It is called on every classification write, tag assignment write, note write, team assignment write, and search operation.
- Files: `backend/app/utils.py`, called from every router that validates a `node_id`
- Impact: For ~2900 nodes the latency is negligible in practice, but it scales linearly with tree size and is called multiple times per request in some paths.
- Fix approach: Build a flat `dict[str, ProcessNode]` lookup on startup and store it in `app.state` alongside the tree.

**Search is O(n) full-tree scan with O(n²) ancestor collection:**
- Issue: `_search_tree` recurses the full tree, and for each matching node calls `_collect_ancestors` which walks the tree again from the root. For a ~2900-node tree with many matches, this is O(n²).
- Files: `backend/app/routers/search.py`
- Impact: Slow on broad queries (e.g. single letter, though `min_length=2` provides some guard). For 2900 nodes typically fast enough, but degrades on common two-letter substrings.
- Fix approach: Pre-compute a parent-map or ancestor-map at startup.

## Known Bugs

**Seeding success count always shows 0:**
- Symptoms: After "Seed with AI" completes, the button shows "Seeded (0 classified)" regardless of how many processes were actually classified.
- Files: `frontend/src/components/SeedingButton.tsx` line 46
- Trigger: Always — the wrong key is read from the result object.
- Workaround: None visible to the user.

**Tag cascade flag stored and compared as string "true"/"false", not boolean:**
- Symptoms: `TagAssignment.cascade` is typed as `str` (not `bool`) in `backend/app/models.py` line 69. Frontend comparisons use `a.cascade === 'true'` (string equality). This is internally consistent but fragile — any code path that serialises cascade as a JSON boolean would silently break filtering.
- Files: `backend/app/models.py` line 69, `frontend/src/components/ProcessDetail.tsx` line 30, `frontend/src/components/ProcessTile.tsx` line 149
- Trigger: Only breaks if cascade is written as a boolean instead of the string "true".
- Workaround: Current write path always uses `str(e.cascade).lower()` in `tags.py` line 103.

## Performance Bottlenecks

**Full classification/tag/team file re-read on every API request:**
- Problem: Every GET `/classifications`, GET `/tags`, GET `/tags/assignments`, GET `/teams` call reads and parses the entire Markdown file from disk. With React Query's `staleTime: 0`, these are re-fetched on every component mount.
- Files: All routers; `frontend/src/hooks/useTags.ts`, `frontend/src/hooks/useClassifications.ts`
- Cause: No in-memory cache; `staleTime: 0` means no client-side cache either.
- Improvement path: Set `staleTime` to 5-30 seconds in hooks; add a simple in-memory cache in Python routers that is invalidated on write.

**`buildNodeMap` rebuilds a full deep map on every `TreeView` render:**
- Problem: `buildNodeMap` in `frontend/src/components/TreeView.tsx` recurses the entire domain subtree on every render. It is wrapped in `useMemo` keyed on `domain`, so it re-runs only when the domain changes — acceptable in practice but memory-intensive for large domains.
- Files: `frontend/src/components/TreeView.tsx` lines 14-24
- Cause: Deep recursive copy into a flat Map.
- Improvement path: Build the map once at the App level from the full process tree and pass it down via context.

## Fragile Areas

**Excel file hard-baked into Docker image:**
- Files: `backend/Dockerfile` line 14
- Why fragile: The Excel file is `COPY`'d into the image at build time. If the file is missing from `docs/` the Docker build fails silently with a missing-file error. The file is gitignored (`docs/` appears in `.gitignore` based on the git history commit `c217dcf fix up tags` and the `.env` note in CLAUDE.md), meaning a fresh clone without the Excel file produces an unbuildable image.
- Safe modification: Ensure `docs/GB921_Business_Process_Framework_Processes_Excel_v25.5.xlsx` is present before running `docker compose build`. Consider adding a `RUN test -f /app/data_source/...` layer to produce a clear error.
- Test coverage: `backend/test_parser.py` requires the file; will fail without it.

**Backend startup hard-fails if Excel is missing:**
- Files: `backend/app/parser.py` lines 46-50, `backend/app/main.py` line 18
- Why fragile: `parse_excel(EXCEL_PATH)` raises `FileNotFoundError` inside `lifespan()`. FastAPI propagates this as an unhandled exception, crashing the process. There is no graceful degradation or helpful log output beyond the Python traceback.
- Safe modification: Wrap `parse_excel` in a try/except in `lifespan()` and log a clear startup error with the expected path.

**Markdown table parser is brittle against pipe characters in cell values:**
- Files: `backend/app/persistence.py` lines 88-99
- Why fragile: `parse_row` splits on `|` after stripping outer pipes. The `write_md_table` escaper replaces `|` with `\|`, but `parse_row` does not un-escape `\|`, so escaped pipes survive as literal `\|` in the parsed string. Notes fields entered with pipe characters (common in Markdown tables) will be corrupted.
- Safe modification: Add `cell.replace('\\|', '|')` in `parse_row`.
- Test coverage: Not tested.

**Chat `_tree_context_cache` is a module-level global, not tied to app state:**
- Files: `backend/app/routers/chat.py` line 17
- Why fragile: In tests or multi-worker deployments, the cache is shared across all app instances in the same process. If tests reload the module between runs, stale context may persist.
- Safe modification: Move the cache into `app.state` alongside `process_tree`.

**`data/` directory contains a copy of the Excel source file:**
- Files: `data/GB921_Business_Process_Framework_Processes_Excel_v25.5.xlsx`
- Why fragile: The `data/` directory is Docker-volume-mounted for persistence of annotation Markdown files. The Excel file is also present there (perhaps placed manually), meaning it could be overwritten or deleted by volume operations. The backend does not use this copy — it uses the Docker-image-baked copy at `/app/data_source/`. Having two copies creates confusion.
- Safe modification: Remove the Excel file from `data/` and rely solely on the image-baked copy; update `.gitignore` to exclude `data/*.xlsx`.

## Missing Critical Features

**No frontend test suite:**
- Problem: Zero test files exist in `frontend/src/`. There is no Jest, Vitest, or Playwright configuration.
- Blocks: Any refactoring of filtering logic (`TreeView.tsx`, `FilterBar.tsx`), store changes (`navigation.ts`, `filters.ts`), or API hook changes carries full regression risk.
- Priority: High — the filter logic in `TreeView.tsx` is complex enough to produce subtle bugs under change.

**No backend integration tests for write paths:**
- Problem: `backend/test_parser.py` tests only the Excel parser. `backend/app/persistence.py` has a self-test `__main__` block but it is not run by any CI harness. There are no tests for any router write operations (tag create/update/delete, classification upsert, note write).
- Files: `backend/test_parser.py`, `backend/app/persistence.py` lines 191-207
- Risk: Silent regressions in the Markdown read-parse-write cycle.

**No CI pipeline:**
- Problem: `docker-compose.yml` has no health-check dependency (`depends_on` does not wait for the backend to be healthy). There is no GitHub Actions or other CI configuration file in the repository.
- Impact: Docker Compose starts the frontend before the backend is ready; the frontend will show "Failed to load process tree" on first load and recover only after React Query retries.
- Fix approach: Add a `healthcheck` block to the backend service and use `condition: service_healthy` in the frontend `depends_on`.

## Scaling Limits

**All annotation data is flat Markdown — no indexing:**
- Current capacity: Works well for hundreds to low thousands of annotated nodes.
- Limit: Markdown table parse time is O(rows); with thousands of classifications and tag assignments, read latency grows linearly. At ~5000+ classifications the per-request parse overhead becomes noticeable.
- Scaling path: Migrate to SQLite (single-file, no server required) for structured annotation data while keeping notes as Markdown files.

**In-memory process tree holds ~2900 nodes per worker:**
- Current capacity: ~2900 nodes, small memory footprint (~few MB).
- Limit: Not a scaling concern for the current dataset; would only become an issue if the eTOM framework itself grew by an order of magnitude.

## Dependencies at Risk

**No pinned dependency versions in `requirements.txt`:**
- Risk: `fastapi>=0.111.0`, `uvicorn[standard]>=0.29.0`, `anthropic>=0.28.0` use `>=` constraints with no upper bound. A breaking major version release (e.g. FastAPI 1.0, Anthropic SDK 1.0) will be silently picked up on the next `pip install`.
- Files: `backend/requirements.txt`
- Impact: Could break the backend on fresh installs without warning.
- Migration plan: Pin to exact versions (`==`) or use a lockfile (e.g. `pip-compile` / `uv`).
